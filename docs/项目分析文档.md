# 项目分析文档

> 基于 Express 的多功能工具箱，集成待办管理（Todo）、天气查询和文本翻译三大功能。
> 生成日期：2026-06-24

---

## 一、项目概览

**项目名称**: my-tool

**项目定位**: 学习练手项目，基于 Node.js + Express 构建的轻量级 Web 工具箱。

**核心功能**:
| 功能模块 | 访问路径 | 说明 |
| -------- | -------- | ---- |
| ✅ 待办管理（Todo） | `GET/POST /todos` | 内存级别的待办增删查（简易 CRUD） |
| 🌤️ 天气查询 | `/api/weather/:city` | 调用 wttr.in 免费 API 查询全球城市天气 |
| 🌐 文本翻译 | `POST /api/translate` | 调用 MyMemory 免费 API 实现多语言互译 |

**启动方式**:

```bash
node app.js
# 访问 http://localhost:3000
```

## 四、代码模块深度分析

### 4.1 入口文件 `app.js`

```javascript
const express = require("express");
const weatherRouter = require("./routes/weather");
const translateRouter = require("./routes/translate");

const app = express();
const port = 3000;

app.use(express.json());       // 解析 JSON 请求体
app.use(express.static("public")); // 托管静态文件

// Todo 功能（内存存储）
let todos = [];               // 内存数组，重启后数据丢失
app.get("/todos", ...);       // 获取所有待办
app.post("/todos", ...);      // 新增待办

// 工具功能路由
app.use("/api/weather", weatherRouter);
app.use("/api/translate", translateRouter);

app.listen(port, () => { ... });
```

**关键分析**:

- **Todo 功能**未使用 Router 模块化，直接在 `app.js` 中定义路由，属于简易实现
- **静态文件托管**在前，**API 路由挂载**在后，顺序合理（避免路由冲突）
- `express.json()` 中间件为全局使用，translate 路由依赖其解析 POST 请求体
- 服务监听 `3000` 端口，硬编码而非环境变量配置

### 4.2 路由层

#### `routes/weather.js` — 天气路由

| 端点                 | HTTP 方法 | 功能                     | 参数来源     |
| -------------------- | --------- | ------------------------ | ------------ |
| `/api/weather/:city` | GET       | 查询指定城市天气         | URL 路径参数 |
| `/api/weather`       | GET       | 查询默认城市天气（北京） | 无参数       |

**代码分析**:

- 使用 Express Router 模块化，符合最佳实践
- 参数校验：空字符串 / 纯空白符检查
- 错误处理：404（城市不存在）、500（服务端异常）两级
- `try-catch` 覆盖完整，异常传播给调用方

#### `routes/translate.js` — 翻译路由

| 端点             | HTTP 方法 | 功能     | 参数来源    |
| ---------------- | --------- | -------- | ----------- |
| `/api/translate` | POST      | 翻译文本 | 请求体 JSON |

**代码分析**:

- 仅一个端点，保持简洁
- 参数校验：文本非空、目标语言必选（源语言可选，默认 `auto`）
- `try-catch` 覆盖完整

### 4.3 服务层

#### `services/weatherService.js` — 天气服务

**调用流程**:

```
getWeatherByCity(city)
  → encodeURIComponent(city)  // URL 编码
  → fetch(`https://wttr.in/${city}?format=j1`)
  → 5s 超时控制 (AbortController)
  → 解析 response.json()
  → 字段提取与转换
  → 返回格式化对象
```

**返回数据格式**:

```javascript
{
  city: "Beijing",          // 城市名（从 API 返回的 areaName 提取）
  country: "China",         // 国家
  temperature: "22",        // 温度（°C）
  feelsLike: "20",          // 体感温度（°C）
  humidity: "45",           // 湿度（%）
  weatherDesc: "Sunny",     // 天气描述
  windSpeed: "15",          // 风速（km/h）
  visibility: "10",         // 能见度（km）
}
```

**关键分析**:

- 使用 **AbortController** 实现 5 秒超时，超时后抛出友好错误
- 对 API 返回字段进行了**重命名映射**（如 `temp_C` → `temperature`，`FeelsLikeC` → `feelsLike`），使字段名更符合前端消费习惯
- 可选链操作符 `?.` 保护空数据访问
- `finally` 中清理超时定时器，防止内存泄漏
- `encodeURIComponent` 处理中文城市名

#### `services/translateService.js` — 翻译服务

**调用流程**:

```
translateText(text, source, target)
  → 参数校验
  → srcLang = source || "auto"  // 默认自动检测
  → tgtLang = target || "zh-CN"
  → fetch(`https://api.mymemory.translated.net/get?q=...&langpair=...`)
  → 5s 超时控制 (AbortController)
  → 验证 responseStatus === 200
  → 返回翻译结果
```

**返回数据格式**:

```javascript
{
  translatedText: "你好，世界",  // 翻译结果
  source: "en",                // 源语言
  target: "zh-CN",             // 目标语言
  match: 100,                  // 匹配度（1-100）
}
```

**关键分析**:

- 内置 `SUPPORTED_LANGUAGES` 常量（16种语言），供前端渲染语言选择下拉框使用
- 同时对 HTTP 状态码和业务状态码（`responseStatus`）做双重校验
- `source || "auto"` 逻辑支持自动检测语言

### 4.4 前端页面

#### `public/index.html` — 工具首页

- **布局**: 居中卡片式布局，渐变背景
- **交互**: 点击卡片通过 `<a>` 标签跳转至对应功能页
- **设计**: 响应式设计（`flex-wrap`），支持移动端

#### `public/weather.html` — 天气查询页面

- **交互流程**: 输入城市 → 点击查询/回车 → `await fetch(/api/weather/城市)` → 渲染结果
- **UI 状态管理**: 加载中（loading）、错误提示（error）、结果展示（weatherResult）三种状态的显隐控制
- **数据绑定**: 通过 `document.getElementById().textContent` 手动绑定数据
- **错误处理**: 展示友好错误信息，请求失败时恢复按钮可用状态

#### `public/translate.html` — 翻译工具页面

- **交互流程**: 选择语言 → 输入文本 → 点击翻译 → `POST /api/translate` → 展示结果
- **特色交互**: 语言交换（`⇄` 按钮）、一键复制结果（Clipboard API）、Ctrl+Enter 快捷键
- **复制功能**: 使用 `navigator.clipboard.writeText()`，带 2 秒反馈恢复
- **交换逻辑**: 当源语言为 `auto`（自动检测）时禁用交换

## 六、数据流分析

### 6.1 天气查询数据流

```
用户输入 "北京"
  → weather.html 的 fetch(`/api/weather/北京`)
  → Express 路由匹配 GET /api/weather/:city
  → weather.js 调用 getWeatherByCity("北京")
  → weatherService.js 发起 HTTP GET wttr.in/北京?format=j1
  → wttr.in 返回 JSON
  → weatherService.js 提取字段并格式化
  → 返回给路由层，包装为 { success: true, data: {...} }
  → 返回 HTTP Response
  → weather.html 解析 JSON，更新 DOM
```

### 6.2 翻译数据流

```
用户选择 en → zh-CN，输入 "Hello"
  → translate.html 的 POST /api/translate
  → Express 解析 JSON 请求体
  → translate.js 调用 translateText("Hello", "en", "zh-CN")
  → translateService.js 发起 HTTP GET MyMemory API
  → MyMemory 返回 JSON
  → translateService.js 提取翻译结果
  → 返回给路由层，包装为 { success: true, data: {...} }
  → 返回 HTTP Response
  → translate.html 解析 JSON，展示结果
```

## 七、项目特点与设计模式分析

### 7.1 优点

| 方面                         | 说明                                                               |
| ---------------------------- | ------------------------------------------------------------------ |
| **模块化清晰**               | 路由/服务/静态资源分离，职责单一                                   |
| **错误处理完整**             | 每一层都有 try-catch，前端也有错误处理                             |
| **超时控制**                 | 两个 Service 均实现了 5 秒请求超时                                 |
| **API 字段解耦**             | Service 层对 API 返回字段进行重命名映射，前端不依赖外部 API 字段名 |
| **零外部依赖（除 Express）** | 充分利用 Node.js 18+ 内置能力                                      |
| **前端体验**                 | 加载状态、错误提示、禁用按钮防止重复提交                           |

### 7.2 可改进点

| 问题               | 建议                                                      |
| ------------------ | --------------------------------------------------------- |
| Todo 功能未模块化  | 提取为独立的 `routes/todo.js` 路由模块                    |
| 端口和 CORS 硬编码 | 使用环境变量（`process.env.PORT`）                        |
| 无日志系统         | 引入 `debug` 或 `pino` 记录请求日志                       |
| 前端无构建工具     | 当前为原生 JS 适合学习，但复杂场景可引入 Vite             |
| 无 TypeScript      | 适合引入类型定义提升可维护性                              |
| 无自动化测试       | 可添加 Jest/Supertest 进行 API 测试                       |
| 内存数据不持久化   | Todo 数据在服务重启后丢失，可引入 SQLite 或 JSON 文件存储 |

### 7.3 使用的设计模式

| 模式           | 体现                                                             |
| -------------- | ---------------------------------------------------------------- |
| **MVC 模式**   | Model（Service 层）— View（public/）— Controller（routes/）      |
| **外观模式**   | Service 层封装复杂的外部 API 调用，对外暴露简洁接口              |
| **中间件模式** | Express 中间件链（`express.json()` → `express.static()` → 路由） |
| **模块模式**   | Node.js 的 `module.exports` / `require` 模块化                   |

## 八、外部 API 风险评估

### 8.1 wttr.in（天气）

| 维度     | 评估                                             |
| -------- | ------------------------------------------------ |
| 可用性   | 稳定，无 API Key 限制，但无 SLA 保障             |
| 速率限制 | 无明显限制，但需合理使用                         |
| 数据覆盖 | 全球主要城市，中文城市名支持良好                 |
| 备选方案 | OpenWeatherMap（需 Key）、WeatherAPI（有限免费） |

### 8.2 MyMemory（翻译）

| 维度     | 评估                                                       |
| -------- | ---------------------------------------------------------- |
| 可用性   | 无 Key 约 1000 次/天，生产环境需注册 Key 提升额度          |
| 翻译质量 | 中英互翻质量良好，长文本翻译稳定                           |
| 备选方案 | Google Translate API（付费）、LibreTranslate（自部署开源） |

## 十一、总结

本项目是一个**功能完整、架构清晰的 Express 练习项目**，覆盖了以下关键开发技能：

1. **Express 5 路由设计** — RESTful API 设计与模块化路由
2. **外部 API 集成** — 免费公共 API 的 HTTP 调用与数据转换
3. **异步编程** — `async/await`、`try-catch` 错误处理、超时控制
4. **前端交互** — 原生 JavaScript 的 fetch API、DOM 操作、状态管理
5. **架构设计** — 三层分离（路由/服务/视图）、模块化导出

项目的核心价值在于以极少的依赖（仅 Express）实现了一个可直接运行的实用工具集，非常适合 Node.js 后端开发入门学习。
